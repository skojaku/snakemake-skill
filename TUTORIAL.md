# Snakemake Tutorial & Reference

## What is Snakemake?

Snakemake is a language that defines workflow. Once you set up a workflow, you can run the whole pipeline from raw data all the way down to figures in your paper by pressing a button. Snakemake automatically parallelizes the workflow and distributes computation.

A workflow consists of *rules*. Each rule takes a file and generates another file. A rule can take files generated by another rule as input, creating links between rules that form a directed network — a *computation graph* — from first input to final output. Snakemake finds an optimal schedule to execute each rule and skips any rule whose output already exists. This is invaluable because we often change parts of a workflow and want to redo only the calculations affected by the change.

Paper: [Sustainable data analysis with Snakemake](https://f1000research.com/articles/10-33/v2)

## Recommended Configuration

Snakemake's defaults prioritize reproducibility over convenience — it reruns scripts when they're updated and halts on any error. During development, I prefer to skip rerunning updated scripts and let Snakemake continue past errors so I can triage while other jobs finish.

Create `~/.config/snakemake/default/config.yaml`:

```yaml
keep-going: True
rerun-triggers: mtime
```

Set the profile:

```bash
echo "export SNAKEMAKE_PROFILE=$HOME/.config/snakemake/default" >> ~/.bashrc
source ~/.bashrc
```

## Kill All Snakemake Jobs

Create `~/.script/stop-snakemake`:

```bash
#!/bin/bash
ps -x | grep $(pwd) | cut -d"?" -f 1 | cut -d"p" -f 1 | xargs kill
ps -x | grep snakemake | grep core | cut -d"?" -f 1 | cut -d"p" -f 1 | xargs kill
```

```bash
mkdir -p ~/.script
chmod +x ~/.script/stop-snakemake
echo "export PATH=\$PATH:\$HOME/.script/" >> ~/.bashrc
source ~/.bashrc
```

Run `stop-snakemake` to kill all Snakemake jobs — useful when you find a bug mid-run.

## Directives: `shell` vs `script`

Both `shell` and `script` define how to generate output from input. `shell` runs a shell command; `script` runs a Python (or R) script with access to the `snakemake` object.

```python
# shell version
rule sort_shell:
    input: input_file = UNSORTED_FILE
    output: output_file = SORTED_FILE
    run: shell("python main.py {input.input_file} {output.output_file}")

# script version
rule sort_script:
    input: input_file = UNSORTED_FILE
    output: output_file = SORTED_FILE
    script: "main.py"
```

I prefer `script` — it's more readable, handles many variables cleanly, and parameters are accessed by name instead of position:

```python
input_file = snakemake.input["input_file"]
output_file = snakemake.output["output_file"]
```

The drawback: scripts aren't standalone since `snakemake` is only defined when run via Snakemake. Fix this with dual-mode:

```python
import sys

if "snakemake" in sys.modules:
    input_file = snakemake.input["input_file"]
    output_file = snakemake.output["output_file"]
else:
    input_file = "../data/sample.npz"
    output_file = "../data/output.csv"
```

Now the script works both in the pipeline and standalone for testing.

## Wildcards

Wildcards abstract out file names so the same template generates different files. This lets you define independent parallel jobs concisely.

Suppose you want 10 simulation runs with different seeds:

```python
SIM_RESULT = "results/run_id~{run_id}.csv"
FIG_SIM_RESULT = "figs/run_id~{run_id}.pdf"
```

The shared wildcard `{run_id}` creates an implicit link:

```python
rule plot:
    input: SIM_RESULT
    output: FIG_SIM_RESULT
    script: "plot.py"

rule all:
    input: expand(FIG_SIM_RESULT, run_id=list(range(10)))
```

Snakemake backtracks from the final products, builds the dependency graph, and runs independent rules in parallel.

## Paramspace

Exploring many parameter combinations gets tedious. Snakemake's `Paramspace` ([docs](https://snakemake.readthedocs.io/en/stable/snakefiles/rules.html#parameter-space-exploration)) helps by generating wildcard patterns from a DataFrame.

This repo's `utils.smk` makes it simpler — no need to create DataFrames manually:

```python
include: "./utils.smk"

params_model = {
    "symmetric": [False, True],
    "aging": [False, True],
    "dim": [16, 64, 128],
}

model_ps = to_paramspace(params_model)

MODEL_FILE = f"model_{model_ps.wildcard_pattern}.pt"
# -> model_symmetric~{symmetric}_aging~{aging}_dim~{dim}.pt
```

Use in rules — wildcards forward parameter values to the script:

```python
rule fit_model:
    input: net_file = CITATION_NET,
    output: output_file = MODEL_FILE
    params:
        dim = lambda wildcards: wildcards.dim,
        symmetric = lambda wildcards: wildcards.symmetric,
        aging = lambda wildcards: wildcards.aging,
    script: "workflow/fitting.py"
```

Access in script:

```python
dim = int(snakemake.params["dim"])
```

## Visualize the Computation Graph

```bash
snakemake --dag all | dot -Tpng > dag.png
```

This generates a DAG showing all task dependencies — useful for understanding the workflow and debugging.
